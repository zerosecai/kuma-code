# JetBrains Session Model ‚Äî Unified Phase Cleanup

## Goal

Collapse the JetBrains `SessionModel`'s three independent state fields (`phase`, `prompt`, `status`) into a single `SessionPhase` sealed class. The UI should see exactly one "what is this session doing?" value at all times. No contradictory combinations possible.

---

## Current Model (Problems)

**`ContentModel.kt:28-33`** ‚Äî State types:

```kotlin
enum class SessionPhase { IDLE, BUSY }

sealed class StatusState {
    data class Thinking(val text: String) : StatusState()
    data class Working(val text: String) : StatusState()
}
```

**`QuestionState.kt:63-75`** ‚Äî Prompt overlay:

```kotlin
sealed class PromptState {
    data class Question(override val id: String, val state: QuestionState) : PromptState()
    data class Permission(override val id: String, val state: PermissionState) : PromptState()
}
```

**`SessionModel.kt:35-42`** ‚Äî Three independent fields:

```kotlin
var phase: SessionPhase = SessionPhase.IDLE      // IDLE or BUSY
var prompt: PromptState? = null                   // Question, Permission, or null
var status: StatusState? = null                   // Thinking, Working, or null
```

### Problems

1. **`phase = BUSY` + `prompt = Permission(...)` is a real combo** ‚Äî but the phase says "busy" when the session is actually blocked waiting for the user. The UI must check both.

2. **`phase = BUSY` + `status = null` can happen** ‚Äî between TurnOpen and the first PartUpdated, status is set but could be cleared/stale from a previous event.

3. **Three fields create combinatorial states** ‚Äî `2 √ó 3 √ó 3 = 18` combos, most invalid. Only ~6 are real.

4. **Permission/Question events are not in `ChatEventDto`** ‚Äî `SessionModel.askQuestion()` and `SessionModel.askPermission()` exist but `SessionManager.handle()` has no cases for them. These features are unwired.

5. **`SessionManager` line 128-136 tracks `session.status` via a Flow BUT also sets phase from TurnOpen/TurnClose** ‚Äî two competing sources of truth for the same field, creating race potential.

---

## Proposed Model

Replace the three fields with a single `SessionPhase` sealed class:

```kotlin
sealed class SessionPhase {
    /** Nothing happening. Ready for user input. */
    data object Idle : SessionPhase()

    /** LLM is actively processing. */
    data class Working(val status: StatusState) : SessionPhase()

    /** Waiting for user to answer a question or approve a permission. */
    data class Prompting(val prompt: PromptState) : SessionPhase()

    /** Request failed, retrying with backoff. */
    data class Retry(val attempt: Int, val message: String, val next: Long) : SessionPhase()

    /** Cannot reach the provider. */
    data class Offline(val requestId: String, val message: String) : SessionPhase()

    /** Turn ended with an error. User can still send a new prompt. */
    data class Error(val message: String, val kind: String? = null) : SessionPhase()
}
```

6 variants. Mutually exclusive. Matches how the CLI behaves ‚Äî when a turn errors, the session stops and the error is the final state of that turn. The user can start a new turn (Error ‚Üí Working), but the session is NOT idle ‚Äî it's visibly in an error state until the user acts.

The existing `ErrorState` data class and the `_errors` accumulated list can be removed from `SessionModel`. When the phase is `Error`, the UI renders the error message from the phase itself. Previous errors are visible in the message history (each assistant message has an `error` field in the CLI). There's no need to track a separate error list in the model.

### What Gets Removed

| Old field                    | Absorbed into                    |
| ---------------------------- | -------------------------------- |
| `phase: SessionPhase` (enum) | The sealed class itself          |
| `prompt: PromptState?`       | `SessionPhase.Prompting(prompt)` |
| `status: StatusState?`       | `SessionPhase.Working(status)`   |
| `_errors: List<ErrorState>`  | `SessionPhase.Error(message)`    |

The `PromptState` and `StatusState` types remain as-is ‚Äî they become data _inside_ the relevant phase variant rather than independent model fields. `ErrorState` is no longer needed as a separate type.

The valid transitions, matching CLI behavior:

| From      | To        | Trigger                                       |
| --------- | --------- | --------------------------------------------- |
| Idle      | Working   | TurnOpen / user sends prompt                  |
| Working   | Idle      | TurnClose (completed)                         |
| Working   | Error     | ChatEventDto.Error                            |
| Working   | Prompting | permission.asked, question.asked              |
| Working   | Retry     | session.status = retry                        |
| Working   | Offline   | session.status = offline                      |
| Prompting | Working   | permission.replied, question.replied          |
| Prompting | Idle      | permission.rejected, question.rejected        |
| Retry     | Working   | retry succeeds (session.status = busy)        |
| Retry     | Idle      | abort during retry                            |
| Offline   | Working   | connectivity restored (session.status = busy) |
| Error     | Working   | TurnOpen / user sends new prompt              |

## Implementation Steps

### Step 1: Update `SessionPhase` type

**File**: `frontend/.../model/ContentModel.kt`

Change `SessionPhase` from a 2-value enum to a sealed class with 6 variants. Keep `StatusState` as-is (it becomes the payload of `Working`). Remove `ErrorState` ‚Äî its data is absorbed into `SessionPhase.Error`.

### Step 2: Remove `prompt` and `status` from SessionModel

**File**: `frontend/.../model/SessionModel.kt`

- Remove `var prompt: PromptState?` field
- Remove `var status: StatusState?` field
- Remove `_errors: MutableList<ErrorState>` field and `errors` getter
- Remove `askQuestion()`, `askPermission()`, `resolvePrompt()`, `setStatus()`, `addError()` methods
- All state changes go through `setPhase()` ‚Äî one setter, one event

Update `loadHistory()` and `clear()` to only reset `phase = Idle`.

### Step 3: Add permission/question events to ChatEventDto

**File**: `shared/.../dto/ChatDto.kt`

Add new event variants:

```kotlin
@SerialName("permission.asked")
data class PermissionAsked(val sessionID: String, val request: PermissionRequestDto) : ChatEventDto()

@SerialName("permission.replied")
data class PermissionReplied(val sessionID: String, val requestID: String) : ChatEventDto()

@SerialName("question.asked")
data class QuestionAsked(val sessionID: String, val request: QuestionRequestDto) : ChatEventDto()

@SerialName("question.replied")
data class QuestionReplied(val sessionID: String, val requestID: String) : ChatEventDto()

@SerialName("question.rejected")
data class QuestionRejected(val sessionID: String, val requestID: String) : ChatEventDto()

@SerialName("session.status")
data class SessionStatus(val sessionID: String, val status: SessionStatusDto) : ChatEventDto()
```

Add corresponding DTOs for `PermissionRequestDto` and `QuestionRequestDto` based on the CLI's `Permission.Request` and `Question.Request` schemas.

### Step 4: Update SessionManager.handle()

**File**: `frontend/.../model/SessionManager.kt`

Update the event handler to drive all transitions through `setPhase()`:

- `TurnOpen` ‚Üí `setPhase(Working(Thinking("Considering...")))`
- `TurnClose` ‚Üí `setPhase(Idle)`
- `Error` ‚Üí `setPhase(Error(message, kind))`
- `PartUpdated` while `phase is Working` ‚Üí `setPhase(Working(status()))` (update status text)
- `PermissionAsked` ‚Üí `setPhase(Prompting(Permission(...)))`
- `PermissionReplied` ‚Üí `setPhase(Working(Thinking("Considering...")))` (resumes)
- `QuestionAsked` ‚Üí `setPhase(Prompting(Question(...)))`
- `QuestionReplied` ‚Üí `setPhase(Working(Thinking("Considering...")))` (resumes)
- `SessionStatus(type=retry)` ‚Üí `setPhase(Retry(...))`
- `SessionStatus(type=offline)` ‚Üí `setPhase(Offline(...))`
- `SessionStatus(type=idle)` ‚Üí `setPhase(Idle)`
- `SessionStatus(type=busy)` ‚Üí `setPhase(Working(...))` only if currently Idle or Error

Remove the `statuses.collect` flow (lines 127-137) ‚Äî session status changes should come through `ChatEventDto.SessionStatus` instead of a separate Flow, keeping one event source.

### Step 5: Simplify SessionModelEvent

**File**: `frontend/.../model/SessionModelEvent.kt`

- Remove `StatusChanged(status)` and `PromptChanged(prompt)` events
- `PhaseChanged(phase)` is the only transition event needed
- The UI reacts to `PhaseChanged` and reads the phase variant to decide what to show

### Step 6: Simplify SessionEvent

**File**: `frontend/.../model/SessionEvent.kt`

- Remove `StatusChanged(text)`, `BusyChanged(busy)`, and `Error(message)` events
- Replace with `PhaseChanged(phase: SessionPhase)`
- UI derives busy from `phase !is Idle && phase !is Error`, status text from `phase` variant
- Errors are visible via `phase is Error` ‚Äî no separate error event needed

### Step 7: Wire up backend SSE parsing

**File**: `backend/.../cli/KiloCliDataParser.kt` (or wherever SSE events are parsed)

Map the raw SSE events (`permission.asked`, `question.asked`, `session.status`, etc.) to the new `ChatEventDto` variants. Currently only `session.turn.open`, `session.turn.close`, `part.updated`, `part.delta`, `message.updated`, `message.removed`, and `error` are handled.

### Step 8: Update SessionUi and other UI consumers

**Files**: `frontend/.../session/SessionUi.kt`, `frontend/.../session/ui/StatusPanel.kt`, `frontend/.../session/ui/PromptPanel.kt`

Switch on a single `phase` value:

```kotlin
when (val phase = chat.phase) {
    is SessionPhase.Idle -> showPromptInput()
    is SessionPhase.Working -> showSpinner(phase.status)
    is SessionPhase.Prompting -> showPromptDock(phase.prompt)
    is SessionPhase.Retry -> showRetryCountdown(phase)
    is SessionPhase.Offline -> showOfflineMessage(phase)
    is SessionPhase.Error -> showError(phase.message)
}
```

### Step 9: Update tests

Update existing tests (`TurnLifecycleTest`, `StatusComputationTest`) to assert on `PhaseChanged` instead of `BusyChanged` + `StatusChanged`. Add new tests:

- Permission asked ‚Üí phase transitions to `Prompting(Permission(...))`
- Permission replied ‚Üí phase transitions back to `Working`
- Question asked ‚Üí phase transitions to `Prompting(Question(...))`
- Question rejected ‚Üí phase transitions to `Idle`
- Error event ‚Üí phase transitions to `Error(message)`
- New prompt from Error state ‚Üí phase transitions to `Working`
- Retry status ‚Üí phase transitions to `Retry`
- Phase is always exactly one variant (no contradictory combos)

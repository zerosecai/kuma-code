# Unified SessionModel Architecture

## Goal

Restructure the JetBrains chat frontend model layer so there is a **single source of truth** called `SessionModel`.

`SessionModel` should own:

- all message content with typed per-part models
- session runtime state (`idle` / `busy`)
- tool execution state (`pending` / `running` / `completed` / `error`)
- prompt-blocking interactive state through a single `PromptState?`
- permission request state
- question request state

UI work is explicitly deferred until the model architecture is in place.

---

## Design Decision

Use a **single `PromptState?`** with sealed variants instead of separate `question` and `permission` fields.

This keeps the state machine explicit and prevents the model from representing impossible UI states such as “active question and active permission at the same time” unless we intentionally expand the architecture later.

Use **typed state objects and enums** instead of loose booleans and nullable strings.

---

## Current Problems

### Split responsibility

- `SessionManager` currently owns runtime/controller state: `busy`, `partType`, `tool`, subscriptions, prompt sending, history loading
- `SessionState` owns raw chat history and workspace/app config state, which creates an unnecessary extra entity
- `MessageListModel` owns a second copy of render-focused message state

This creates two competing models:

1. `SessionState` as session truth for messages
2. `MessageListModel` as UI truth for messages, status, questions

### Naming mismatch

- `SessionManager` is actually doing controller + derived state work
- `MessageListModel` is closer to the model you want to become `SessionModel`
- `SessionState` duplicates responsibilities that should belong to `SessionModel`

### Untyped prompt/request state

- questions were added as `question: QuestionState?`
- permissions do not exist yet
- status is still modeled as `status: String?`
- runtime turn state is only `TurnState`, but prompt-blocking flow is not unified

---

## Target Architecture

### High-level split

Keep **orchestration** and **model** separate while preserving the current `SessionManager` name:

- `SessionManager`
  - remains the orchestration class
  - owns RPC subscriptions/actions
  - converts CLI/RPC DTOs into `SessionModel` mutations
  - contains no authoritative session state except coroutine/job wiring

- `SessionModel` (rename from current `MessageListModel`)
  - owns the full frontend session state used by the chat UI
  - publishes `SessionModelEvent`
  - becomes the only model the future UI should render from
  - directly handles message add/remove/update, delta streaming, history loading, phase changes, prompt state, permissions, questions, errors, and status

There is **no `SessionState` / `SessionCache` / extra backing entity** in the target architecture.

The key point: after the refactor, `SessionManager` mutates `SessionModel` directly and the UI reads only `SessionModel`.

## Core Model Shape

### `SessionModel` (renamed from `MessageListModel`)

```kotlin
class SessionModel {
    val messages: Collection<MessageModel>

    var phase: SessionPhase
        private set

    var prompt: PromptState?
        private set

    var status: StatusState?
        private set

    val errors: List<ErrorState>

    fun addMessage(info: MessageInfo)
    fun removeMessage(id: String)
    fun updateContent(messageId: String, dto: PartDto)
    fun appendDelta(messageId: String, partId: String, delta: String)
    fun loadHistory(history: List<MessageWithPartsDto>)

    fun setPhase(phase: SessionPhase)
    fun setStatus(state: StatusState?)
    fun addError(err: ErrorState)

    fun askQuestion(state: QuestionState)
    fun askPermission(state: PermissionState)
    fun resolvePrompt(id: String)

    fun clear()
}
```

`SessionModel` directly absorbs the current responsibilities split across `MessageListModel` and `SessionState`.

### Session-level state types

```kotlin
enum class SessionPhase {
    IDLE,
    BUSY,
}

sealed class StatusState {
    data class Thinking(val text: String) : StatusState()
    data class Working(val text: String) : StatusState()
}

data class ErrorState(
    val message: String,
    val kind: String? = null,
)
```

Notes:

- `SessionPhase` replaces ad-hoc `busy: Boolean`
- `StatusState?` replaces raw `status: String?`
- `ErrorState` replaces raw `List<String>` and gives room for typed error handling later

## Message Models

### `MessageModel`

Keep a typed message object with ordered parts.

```kotlin
class MessageModel(
    val info: MessageInfo,
) {
    val parts: LinkedHashMap<String, ContentModel>
}
```

### `MessageInfo`

Introduce a frontend/domain message info model instead of exposing DTOs directly as the long-term model surface.

```kotlin
data class MessageInfo(
    val id: String,
    val role: String,
    val created: Double,
    val completed: Double? = null,
    val agent: String? = null,
    val providerId: String? = null,
    val modelId: String? = null,
    val parentId: String? = null,
    val cost: Double? = null,
)
```

This can be built from `MessageDto` by the `SessionManager`/model boundary.

`SessionModel` owns conversion from DTO-shaped history/event data into domain message state. `SessionManager` forwards DTOs in, but does not store them.

## Content Models

The user asked that “everything we put into session model has its own models, such as Text, Reasoning, Question & etc”.

To make that explicit, split the current `PartModel` and prompt-blocking models under a common naming scheme.

### Message content inside messages

```kotlin
sealed class ContentModel(val id: String) {
    class Text(id: String) : ContentModel(id) {
        val content = StringBuilder()
    }

    class Reasoning(id: String) : ContentModel(id) {
        val content = StringBuilder()
    }

    class Tool(id: String, val name: String) : ContentModel(id) {
        var state: ToolExecState = ToolExecState.PENDING
        var title: String? = null
    }

    class Compaction(id: String) : ContentModel(id)
}

enum class ToolExecState {
    PENDING,
    RUNNING,
    COMPLETED,
    ERROR,
}
```

Notes:

- Rename current `PartModel` to `ContentModel`
- Rename current `ToolState` to `ToolExecState` to avoid confusion with future permission/question state

### Prompt-blocking interaction state

```kotlin
sealed class PromptState {
    abstract val id: String

    data class Question(
        override val id: String,
        val state: QuestionState,
    ) : PromptState()

    data class Permission(
        override val id: String,
        val state: PermissionState,
    ) : PromptState()
}
```

This becomes the session model’s authoritative prompt gate.

## Permission Models

Based on the VS Code model and CLI schema, add typed permission models now even before UI work.

### Core permission state

```kotlin
data class PermissionState(
    val id: String,
    val sessionId: String,
    val name: String,
    val patterns: List<String>,
    val always: List<String>,
    val meta: PermissionMeta,
    val message: String? = null,
    val tool: ToolCallRef? = null,
    val state: PermissionRequestState = PermissionRequestState.PENDING,
)

enum class PermissionRequestState {
    PENDING,
    RESPONDING,
    RESOLVED,
    ERROR,
}
```

### Tool association

```kotlin
data class ToolCallRef(
    val messageId: String,
    val callId: String,
)
```

### Permission metadata

Start with a generic container, but typed enough to avoid `Map<String, Any>` as the final model surface.

```kotlin
data class PermissionMeta(
    val rules: List<String> = emptyList(),
    val diff: String? = null,
    val filePath: String? = null,
    val fileDiff: PermissionFileDiff? = null,
    val raw: Map<String, String> = emptyMap(),
)

data class PermissionFileDiff(
    val file: String,
    val patch: String? = null,
    val before: String? = null,
    val after: String? = null,
    val additions: Int,
    val deletions: Int,
)
```

### Permission response intent

This is useful even before UI implementation.

```kotlin
enum class PermissionReply {
    ONCE,
    ALWAYS,
    REJECT,
}
```

### Why model permission state now

VS Code effectively tracks permissions using:

- pending queue
- responding set
- resolution/error events

For JetBrains, since you want a true state machine and single point of truth, encode that lifecycle directly on the model rather than split it across multiple collections.

## Events

Rename `MessageModelEvent` to `SessionModelEvent` once the rename happens.

```kotlin
sealed class SessionModelEvent {
    data class MessageAdded(val info: MessageInfo) : SessionModelEvent()
    data class MessageRemoved(val id: String) : SessionModelEvent()

    data class ContentAdded(val messageId: String, val content: ContentModel) : SessionModelEvent()
    data class ContentUpdated(val messageId: String, val content: ContentModel) : SessionModelEvent()
    data class ContentDelta(val messageId: String, val contentId: String, val delta: String) : SessionModelEvent()

    data class PhaseChanged(val phase: SessionPhase) : SessionModelEvent()
    data class StatusChanged(val status: StatusState?) : SessionModelEvent()
    data class PromptChanged(val prompt: PromptState?) : SessionModelEvent()
    data class ErrorAdded(val err: ErrorState) : SessionModelEvent()

    data object HistoryLoaded : SessionModelEvent()
    data object Cleared : SessionModelEvent()
}
```

Key change:

- instead of separate `QuestionAsked` / `QuestionResolved` and future `Permission...` events, use one authoritative `PromptChanged(prompt)` event because prompt state is single-source and exclusive

This is a cleaner state-machine API.

## Renames

### Rename `MessageListModel` → `SessionModel`

This is the user-requested rename, but it should come with the responsibility transfer described above.

### Keep `SessionManager`

Do not rename `SessionManager`.

It remains the orchestration class:

- `SessionManager` = orchestration / RPC / subscriptions
- `SessionModel` = state

### Remove `SessionState`

`SessionState` should be deleted, not renamed.

Reason:

- the user explicitly does not want an extra entity
- `SessionModel` should be the point of truth
- message add/remove/update/history logic should live in `SessionModel`
- runtime session state should live in `SessionModel`
- prompt state should live in `SessionModel`

If temporary helpers are needed during the refactor, keep them as private conversion functions inside `SessionManager` or `SessionModel`, not as a long-lived separate class.

## Migration Plan

### Phase 1: Naming + model consolidation

1. Rename current `MessageListModel` to `SessionModel`
2. Keep `SessionManager` name and narrow its responsibilities to orchestration only
3. Rename current `MessageModelEvent` to `SessionModelEvent`
4. Rename `PartModel` to `ContentModel`
5. Rename `ToolState` to `ToolExecState`
6. Delete `SessionState` and move its responsibilities into `SessionModel`

This is mostly mechanical, but important before deeper changes.

### Phase 2: Expand `SessionModel` to true source of truth

Move the following session-facing state into `SessionModel`:

- `phase` / busy-idle runtime state
- `status`
- `errors`
- active prompt state
- typed messages / contents
- message history loading from `MessageWithPartsDto`
- message/part lookup APIs currently served by `SessionState`

`SessionManager` should mutate only `SessionModel`; the UI should read only `SessionModel`.

### Phase 3: Add typed prompt models

1. Replace `question: QuestionState?` with `prompt: PromptState?`
2. Add `PermissionState`, `PermissionMeta`, `PermissionFileDiff`, `PermissionReply`, `ToolCallRef`
3. Add mutations:

```kotlin
fun setPrompt(state: PromptState?)
fun askQuestion(state: QuestionState)
fun askPermission(state: PermissionState)
fun resolvePrompt(id: String)
fun failPrompt(id: String)
```

Recommended actual internal API:

- keep `setPrompt` private
- expose `askQuestion`, `askPermission`, `resolvePrompt`, `setPermissionState`

### Phase 4: Remove remaining `SessionState` usage

At that point `SessionManager` should stop making the UI read back through former cache-style lookups like:

- `model.chat.part(...)`
- `model.chat.message(...)`

Instead, `SessionManager` should transform DTOs directly into `SessionModel` mutations.

Target:

- no `SessionState`
- no `SessionCache`
- no second backing entity behind `SessionModel`

## Permission State Machine

Since you explicitly want pending/running/question/permission state represented in the model, this is the recommended permission lifecycle.

### States

```kotlin
enum class PermissionRequestState {
    PENDING,
    RESPONDING,
    RESOLVED,
    ERROR,
}
```

### Transitions

```text
null
  -> PENDING        on permission.asked
PENDING
  -> RESPONDING     when user submits response
RESPONDING
  -> RESOLVED       on permission.replied
RESPONDING
  -> ERROR          if reply RPC fails
ERROR
  -> RESPONDING     retry
RESOLVED
  -> null           when prompt is dismissed from model/UI flow
```

Because `PromptState` is exclusive, resolved prompts can often transition directly to `null` rather than lingering in a visible resolved state.

---

## Question State Machine

```text
null
  -> Question(PENDING)    on question.asked
Question(PENDING)
  -> Question(RESPONDING) when user submits/rejects
Question(RESPONDING)
  -> null                 on question.replied/question.rejected
Question(RESPONDING)
  -> Question(ERROR)      on reply/reject RPC failure
Question(ERROR)
  -> Question(RESPONDING) retry
```

This suggests `QuestionState` may also need a lifecycle field, not just payload.

Recommended update:

```kotlin
enum class QuestionRequestState {
    PENDING,
    RESPONDING,
    ERROR,
}
```

Then:

```kotlin
data class QuestionState(
    val id: String,
    val items: List<QuestionItem>,
    val tool: ToolCallRef? = null,
    val state: QuestionRequestState = QuestionRequestState.PENDING,
)
```

## Model-First Test Plan

Do this before UI work.

### `SessionModelTest.kt` (renamed from current `MessageListModelTest.kt`)

#### Naming and shape

- rename the test file/class to `SessionModelTest`
- keep it lightweight (`UsefulTestCase`)

### Content/message tests

- message add/remove still work after rename
- `updatePart` creates `ContentModel.Text`

- initial phase is `IDLE`
- `setPhase(BUSY)` updates state and fires event
- `setStatus(Thinking(...))` stores typed status and fires event
- clear resets phase/status/errors/messages/prompt

### Question prompt tests

- `askQuestion` sets `prompt` to `PromptState.Question`
- asking a second question replaces the existing prompt
- `resolvePrompt(id)` clears matching question prompt
- wrong id is noop
- question lifecycle can move `PENDING -> RESPONDING -> ERROR`

### Permission prompt tests

- `askPermission` sets `prompt` to `PromptState.Permission`
- permission fields (`patterns`, `always`, `tool`, `meta`) are preserved
- permission lifecycle can move `PENDING -> RESPONDING -> ERROR`
- `resolvePrompt(id)` clears matching permission prompt

### Exclusivity tests

- asking a permission replaces a question prompt
- asking a question replaces a permission prompt
- prompt state is always at most one variant

### Event tests

- `PromptChanged(prompt)` fires on question set
- `PromptChanged(prompt)` fires on permission set
- `PromptChanged(null)` fires on resolve

## Recommendation

Implement this in **two commits/phases** even if done in one branch:

1. **Pure model rename + state-shape refactor**
2. **SessionManager rewiring to the new model API**

That keeps the migration understandable and makes review significantly easier.
